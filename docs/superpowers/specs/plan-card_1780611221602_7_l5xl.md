## Overview

Integrate the Streamlit dashboard (`app.py`) directly with the headless LinkedIn scraper (`scraper.py`) by isolating the async Playwright loop in a dedicated background thread. The problem: Streamlit reruns top-to-bottom on every interaction, but Playwright's async operations require sustained context across multiple user interactions. Solution: wrap the entire scraper lifecycle (browser launch, profile traversal, data collection) inside a `threading.Thread` that runs independently, communicating state and results back to Streamlit via a thread-safe queue.

The integration unfolds in three layers:
1. **Background worker**: Isolated `threading.Thread` running the Playwright loop with explicit OS environment mappings for Chromium stability on Linux.
2. **Message broker**: `queue.Queue` relaying status updates, warnings, and collected profiles from worker to UI in real time.
3. **UI freeze layer**: Interactive inputs and buttons disabled while collection is active, preventing thread collision.

## Architecture Decisions

### 1. Threading Model

**Decision**: Use `threading.Thread` (not `asyncio.run_in_executor`).

**Why**:
- Streamlit is synchronous; async-in-executor couples Playwright async errors to Streamlit's event loop.
- `threading.Thread` decouples Playwright's event loop fully; exceptions in the worker don't crash Streamlit.
- Explicit thread join in UI gives user control over cancellation.

**Structure**:
```
┌─────────────────────┐
│  Streamlit App      │ (main thread, re-runs on interaction)
│  - UI freeze logic  │
│  - Queue consumer   │ ← reads status updates continuously
│  - Thread spawner   │
└──────────┬──────────┘
           │
           │ threading.Thread
           ↓
┌──────────────────────┐
│  Background Worker   │ (separate OS thread, Playwright runs here)
│  - Playwright loop   │
│  - Profile traversal │
│  - Status emitter    │ → sends updates via Queue
└──────────────────────┘
```

### 2. Queue-Based Message Protocol

**Decision**: `queue.Queue` with structured message objects (not raw strings).

**Why**:
- Thread-safe by design; no explicit locks needed for producer/consumer.
- Structured messages prevent parsing errors and enable versioning.
- Supports multiple message types: status, profile, warning, error, complete.

**Message Types** (defined in `queue_protocol.py`):

```python
# Base message class (all inherit)
class Message:
    timestamp: float  # Unix time; enables ordering
    worker_id: str    # Worker thread ID; future: multi-worker support

# Status update during traversal
class StatusUpdate(Message):
    profile_url: str
    status: str  # "loading", "parsing", "stored"
    elapsed_sec: float

# Collected profile (new payload)
class ProfilePayload(Message):
    profile_data: dict  # Full scraped profile
    url: str

# Operational warning (rate limit approaching, CAPTCHA near)
class OperationWarning(Message):
    severity: str  # "info", "warning", "critical"
    message: str
    action: Optional[str]  # "continue", "pause", "shutdown"

# Execution complete or error halt
class ExecutionComplete(Message):
    success: bool
    profiles_collected: int
    total_queued: int
    error_type: Optional[str]  # ExecutionWindowExceeded, CaptchaDetected, etc.
    details: dict  # Error context
```

### 3. Thread-Safe State Management

**Decision**: Streamlit `st.session_state` + `threading.Lock` for shared resource access.

**Why**:
- Streamlit session_state is isolated per user; thread-safe via Python's GIL for dict reads.
- Lock used only for mutable collections (URLs queue, collected profiles list).
- No explicit locking needed for Queue operations (thread-safe by design).

**Session State Keys**:
```python
st.session_state.scraper_thread: Optional[threading.Thread]
st.session_state.message_queue: queue.Queue
st.session_state.collection_active: bool  # UI freeze trigger
st.session_state.collected_profiles: list  # Real-time accumulation
st.session_state.status_log: list  # Last 50 status messages (for UI display)
st.session_state.current_warning: Optional[OperationWarning]
st.session_state.thread_lock: threading.Lock  # Guard URLs queue, profiles list
```

### 4. OS Environment Isolation

**Decision**: Explicitly map Chromium environment at thread spawn, not at global scope.

**Why**:
- Streamlit's re-run model may init `scraper.CHROMIUM_LAUNCH_ENV` multiple times; global state is unreliable.
- Pass environment as argument to background worker; thread inherits it cleanly.
- Supports future multi-worker deployments (each thread gets its own env copy).

**Implementation**: Refactor `scraper.py` line 43–46 into `environment_config.py`:
```python
def get_chromium_environment():
    """Return Linux Chromium environment mapping."""
    return {
        "DISPLAY": os.environ.get("DISPLAY", ":99"),
        "PATH": os.environ.get("PATH", ""),
        # Add: LD_LIBRARY_PATH, HOME, TMPDIR for stability
    }
```

### 5. UI Freeze Strategy

**Decision**: Disable input widgets conditionally during collection; show live progress bar.

**Why**:
- Prevents user queuing new URLs while thread is running (would race on the URLs queue).
- Live progress bar shows collection is ongoing (Streamlit's re-run paradigm hides background work otherwise).

**Implementation** (in `app.py`):
```python
if st.session_state.collection_active:
    st.warning("⏸️ Collection in progress. Inputs frozen.")
    # Disable button/text input via disabled=True param
    # Show live metrics: profiles collected, current URL, time elapsed

else:
    # Normal UI: text area, button enabled
```

### 6. Collector Worker Entry Point

**Decision**: Expose `async def scrape_profile_batch(urls, queue)` from `scraper.py`.

**Why**:
- Current `scraper.py` has per-profile `scrape_profile()` and helper functions, but no batch entry point.
- Batch function handles: spawn browser, iterate URLs, emit StatusUpdate per profile, collect results in queue, cleanup on error/complete.
- Background thread calls this via `asyncio.run()`.

**Signature**:
```python
async def scrape_profile_batch(
    urls: list[str],
    message_queue: queue.Queue,
    window_controller: ExecutionWindowController,
    anomaly_detector: AnomalyDetector,
    env: dict = None,
) -> None:
    """
    Traverse LinkedIn profiles, emit status updates and payloads to queue.
    Handles browser lifecycle, error recovery, and graceful shutdown.
    """
```

## State Management Strategy

### 1. Session State Flow

**Initialization** (app start):
```
app.py: init → session_state keys all set to defaults
    ├─ scraper_worker: None
    ├─ collection_active: False
    ├─ collected_profiles: []
    └─ status_log: []
```

**Collection Active**:
```
User clicks "Queue for Scraping"
    ↓
ScraperWorker.spawn(urls, env)  ← thread starts
    ↓
app.py marks collection_active = True
    ↓
Background thread emits messages → queue
    ↓
app.py message consumer loop pulls messages
    │
    ├─ StatusUpdate → append to status_log
    ├─ ProfilePayload → append to collected_profiles (under lock)
    ├─ OperationWarning → display in UI
    └─ ExecutionComplete → mark collection_active = False
```

**Collection Complete**:
```
ExecutionComplete received
    ↓
app.py marks collection_active = False
    ↓
Inputs re-enabled
    ↓
User can queue new URLs or view collected_profiles
```

### 2. Data Consistency Guarantees

| Resource | Lock Type | Why |
|----------|-----------|-----|
| `collected_profiles` list | `threading.Lock` | Multiple ProfilePayloads appended from worker; reads from UI |
| `message_queue` | `queue.Queue` | Thread-safe; no explicit lock needed |
| `status_log` list | None (append-only, GIL protected) | Logs are informational; no correctness issue if slightly out-of-order |
| Session state dict | GIL (Python 3) | Dict reads are atomic; no explicit lock needed for simple assignments |

### 3. Error State Propagation

If worker encounters error (e.g., `ExecutionWindowExceeded`):
```
scraper.py: raises exception
    ↓
thread_manager.py: catches in try/except
    ↓
Emits: OperationWarning(severity="critical", action="shutdown")
       ExecutionComplete(success=False, error_type="ExecutionWindowExceeded")
    ↓
app.py: displays warning, marks collection_active = False
    ↓
User can view error log, re-queue with different URLs
```

## Risks and Edge Cases

### 1. Thread Resource Leaks

**Risk**: Background thread continues consuming memory after error; orphan Chromium process persists.

**Mitigation**:
- Wrap browser/context in `try/finally` with `safe_browser_cleanup()` (already in scraper.py:207).
- In `thread_manager.py`, fallback: if thread doesn't exit within timeout, call `psutil.Process.terminate()`.
- Add Linux-specific cleanup: `pkill -f 'chrome.*headless'` as last resort (in comments, not auto-called).

### 2. Queue Overflow

**Risk**: If UI doesn't consume messages fast enough, queue grows unbounded; memory pressure.

**Mitigation**:
- Cap queue size: `queue.Queue(maxsize=1000)` (constructor in `thread_manager.py`).
- If queue full, log warning and drop oldest message (implement `queue_protocol.DropMessage`).
- Monitor in tests: assert queue never exceeds 100 messages during normal operation.

### 3. Rerun Loops

**Risk**: `st.rerun()` in message consumer loop causes infinite re-runs; browser tab becomes unresponsive.

**Mitigation**:
- Only call `st.rerun()` if new message received; add `time.sleep(0.01)` to avoid tight loop.
- Limit rerun frequency: at most 1 rerun per 0.1 sec (use `time.time()` check).
- Test: spawn worker, monitor rerun count over 5 sec; assert < 100 reruns.

### 4. Concurrent URL Queue Access

**Risk**: User modifies `bulk_urls` in text_area while worker reads queue; race condition on parsing.

**Mitigation**:
- URL parsing happens in main thread before spawn; worker receives immutable list.
- Queue is not mutable after spawn (no code path adds URLs mid-collection).
- Lock is defensive (guard `collected_profiles` list only).

### 5. Session State Serialization

**Risk**: Streamlit session_state serializes to Pickle; objects like `threading.Thread` or `queue.Queue` may not serialize.

**Mitigation**:
- Store only serializable objects in session_state: lists, dicts, strings.
- Keep `scraper_worker` and `message_queue` as instance attributes of a serializable wrapper class.
- Alternative: use `st.cache_resource` for worker lifecycle (not recommended; less control).

### 6. Browser Context Inheritance

**Risk**: Playwright async context doesn't transfer to new thread; loop errors occur.

**Mitigation**:
- Don't reuse browser from main thread; spawn fresh in background thread.
- Batch function `scrape_profile_batch()` always calls `launch_stealth_browser()` fresh.
- Test: verify browser URL matches expected profile URL before data extraction.

### 7. Multiple Collections in Parallel

**Risk**: User clicks "Queue" twice; two threads both modify `collected_profiles`.

**Mitigation**:
- Disable "Queue" button while `collection_active=True` (already in UI freeze logic).
- Assert in tests: only 1 worker alive at any time.
- If edge case (e.g., session reload), kill existing worker before spawning new one:
  ```python
  if st.session_state.scraper_worker and st.session_state.scraper_worker.is_alive():
      st.session_state.scraper_worker.terminate()
  ```

### 8. Database Transaction Isolation

**Risk**: Worker inserts profiles into SQLite concurrently with UI reading profiles; dirty reads.

**Mitigation**:
- SQLite WAL mode ensures readers don't block writers; no isolation issue.
- Refresh `load_profile_data()` cache after collection completes:
  ```python
  if execution_complete:
      st.cache_data.clear()  # Clear caching key
      df = load_profile_data(DB_PATH)  # Force reload
  ```

### 9. Linux Chromium DISPLAY Unset

**Risk**: `DISPLAY` env var not set; Chromium fails to launch on headless Linux.

**Mitigation**:
- `environment_config.py` defaults to `:99` (Xvfb).
- Document requirement: "Deploy requires `Xvfb` or `DISPLAY` set."
- Add startup check: if DISPLAY not found and no Xvfb, emit warning (not fatal; Chromium may still work with `--no-sandbox`).

### 10. Thread Cleanup on Streamlit Restart

**Risk**: User refreshes browser; Streamlit server restarts; background thread orphaned.

**Mitigation**:
- On app startup, check for any zombie threads from prior session.
- Use session-unique IDs (worker_id in Message class) to distinguish workers.
- Add timeout to thread join on app exit (handled by OS when Python process terminates).
