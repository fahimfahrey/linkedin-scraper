## Overview

Build `database.py` ŌĆö the local persistence engine for scraped LinkedIn profile data. The module wraps Python's standard-library `sqlite3` to create and manage a `linkedin_profiles` table, protect against duplicate URL inserts during high-volume scrape loops, and expose a Pandas `DataFrame` interface for downstream analysis.

Three deliverables:

1. **Schema initializer** ŌĆö creates `linkedin_profiles` table idempotently (`CREATE TABLE IF NOT EXISTS`)
2. **Insert interface** ŌĆö writes one profile dict using `INSERT OR IGNORE` for 100% duplicate protection
3. **DataFrame utility** ŌĆö reads full table into a `pd.DataFrame` via `pd.read_sql_query`

## Architecture Decisions

### A. SQLite via stdlib `sqlite3` (no ORM)

**Decision**: Use `sqlite3` directly; no SQLAlchemy or third-party ORM.

**Rationale**: Project `requirements.txt` declares no ORM; the task explicitly says "standard Python SQLite primitives." SQLite covers single-process, single-machine scraping. No migration tooling needed at this scope.

**Details**: All DB calls use the `with sqlite3.connect(db_path) as conn:` context manager pattern, which auto-commits on clean exit and rolls back on exception.

---

### B. `INSERT OR IGNORE` for duplicate protection

**Decision**: Use `INSERT OR IGNORE INTO linkedin_profiles (...)` rather than upsert/merge logic.

**Rationale**: Task requires 100% duplicate suppression ŌĆö identical `linkedin_url` on second insert must be silently dropped, not updated. `INSERT OR IGNORE` leverages the `UNIQUE` constraint on `linkedin_url` and is a single atomic statement with no read-before-write race.

**Details**: `conn.execute()` returns a `Cursor`; `cursor.rowcount == 1` means inserted, `cursor.rowcount == 0` means ignored. The function returns this boolean for callers to log/track.

---

### C. Pandas `pd.read_sql_query` for DataFrame extraction

**Decision**: Use `pd.read_sql_query("SELECT * FROM linkedin_profiles", conn)` with an open `sqlite3.Connection`.

**Rationale**: `pd.read_sql_query` is the canonical pandas API for SQL ŌåÆ DataFrame conversion. Keeps the DB connection lifecycle inside `database.py`; caller receives a clean DataFrame with no raw cursor handling.

**Details** (per `pandas-pro` skill):
- Return type is `pd.DataFrame`; caller must use `.copy()` if mutating subsets
- `collected_at` column will be `object` dtype by default; caller can parse with `pd.to_datetime()` after retrieval (not forced inside this module to avoid coupling)
- Empty table returns a valid empty DataFrame (correct columns, zero rows)

---

### D. Module-level `DB_PATH` default

**Decision**: `DB_PATH = "linkedin_profiles.db"` as a module constant; all three functions accept an optional `db_path` override.

**Rationale**: Enables tests to pass an in-memory path (`:memory:`) or a temp file path, keeping unit tests isolated and fast without mocking `sqlite3`.

### Step 2 ŌĆö `init_db(db_path)`

```python
def init_db(db_path: str = DB_PATH) -> None:
    with sqlite3.connect(db_path) as conn:
        conn.execute("""
            CREATE TABLE IF NOT EXISTS linkedin_profiles (
                id              INTEGER PRIMARY KEY,
                linkedin_url    TEXT UNIQUE,
                full_name       TEXT,
                headline        TEXT,
                location        TEXT,
                current_company TEXT,
                about_text      TEXT,
                experience_json TEXT,
                education_json  TEXT,
                collected_at    TIMESTAMP DEFAULT CURRENT_TIMESTAMP
            )
        """)
        conn.commit()
    logger.info("DB initialised at %s", db_path)
```

Key points:
- `CREATE TABLE IF NOT EXISTS` ŌĆö idempotent, safe to call on every startup
- `id INTEGER PRIMARY KEY` ŌĆö auto-increment rowid alias in SQLite
- `linkedin_url TEXT UNIQUE` ŌĆö the UNIQUE constraint that drives `INSERT OR IGNORE`
- `collected_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP` ŌĆö server-side default, no Python `datetime` needed

### Step 3 ŌĆö `insert_profile(profile, db_path)`

```python
def insert_profile(profile: dict, db_path: str = DB_PATH) -> bool:
    sql = """
        INSERT OR IGNORE INTO linkedin_profiles
            (linkedin_url, full_name, headline, location, current_company,
             about_text, experience_json, education_json)
        VALUES
            (:linkedin_url, :full_name, :headline, :location, :current_company,
             :about_text, :experience_json, :education_json)
    """
    with sqlite3.connect(db_path) as conn:
        cursor = conn.execute(sql, profile)
        conn.commit()
        inserted = cursor.rowcount == 1
    if inserted:
        logger.debug("Inserted profile: %s", profile.get("linkedin_url"))
    else:
        logger.debug("Duplicate ignored: %s", profile.get("linkedin_url"))
    return inserted
```

Key points:
- Named `:param` binding prevents SQL injection and handles dict input cleanly
- `INSERT OR IGNORE` ŌĆö duplicate `linkedin_url` is silently dropped at the DB level
- `collected_at` is omitted from the INSERT so it uses `DEFAULT CURRENT_TIMESTAMP`
- `rowcount` check provides the caller a truthful insertion signal

### Step 4 ŌĆö `get_profiles_df(db_path)`

Per `pandas-pro` skill:

```python
def get_profiles_df(db_path: str = DB_PATH) -> pd.DataFrame:
    with sqlite3.connect(db_path) as conn:
        df = pd.read_sql_query("SELECT * FROM linkedin_profiles", conn)
    logger.debug("Loaded %d profiles from %s", len(df), db_path)
    return df
```

Key points:
- `pd.read_sql_query` takes an open connection ŌĆö no separate cursor needed
- Returns empty DataFrame (correct schema, zero rows) when table is empty
- Caller is responsible for `.copy()` if mutating subsets (per pandas-pro constraint)
- `collected_at` comes back as `object` dtype ŌĆö downstream callers use `pd.to_datetime()`

## State Management Strategy

No in-memory state lives in `database.py`. All state is on-disk in the SQLite file. Each function opens and closes its own connection via context manager ŌĆö there is no module-level connection singleton. This ensures:

- No connection leaks between scraper loop iterations
- Tests can pass `:memory:` or a fresh temp path for full isolation
- Thread safety: SQLite in WAL mode can handle concurrent readers; single-writer pattern is safe for the scraper's sequential loop

## Risks and Edge Cases

| Risk | Mitigation |
|------|-----------|
| `database.py` receives a profile dict with extra keys not in the schema | Named `:param` binding ignores unknown keys ŌĆö no error |
| `database.py` receives a profile dict missing optional keys | Named binding maps missing keys to `None`; SQLite stores NULL ŌĆö safe |
| `experience_json` / `education_json` passed as Python list instead of string | Caller must `json.dumps()` before passing ŌĆö `insert_profile` does NOT serialize; document in docstring |
| DB file on read-only filesystem | `sqlite3.OperationalError` propagates to caller ŌĆö not swallowed |
| Concurrent writes from multiple processes | Not in scope; single-process scraper. If parallelised later, switch DB_PATH to WAL mode via `PRAGMA journal_mode=WAL` |
| `get_profiles_df` on large table (100k+ rows) | In-memory load; acceptable for scraper scale. Add `LIMIT`/pagination if needed |
| `pandas` not installed | `ImportError` on `import pandas` ŌĆö fail fast; already in `requirements.txt` |
