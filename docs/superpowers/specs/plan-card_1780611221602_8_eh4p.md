## Architecture Decisions

### 1. Documentation Structure (Hierarchical with Clear Audience Segmentation)
- **Why:** Different audiences need different depths. New developers skip ops sections; ops teams skip architecture deep-dives. Flat docs force everyone to read everything.
- **How:** Use H2 headers for audience/phase (Setup, Operations, Schema) and H3 for details within each phase.

### 2. Playwright + Streamlit + SQLite as Primary Technology Narrative
- **Why:** These three form the core "stack" and are the first thing users need to understand.
- **How:** Lead with this triad in overview, then document each layer's configuration separately.

### 3. PLAYWRIGHT_HOST_PLATFORM_OVERRIDE Prominently in Ubuntu 26.04 Section
- **Why:** This is a critical gotcha (Playwright auto-selects `ubuntu22.04-x64` on Ubuntu 26.04, causing driver mismatch). New developers waste 1-2 hours debugging.
- **How:** Dedicated subsection with explicit `export` command, warning box, and validation test.

### 4. Session.json as Non-Committal Credential Store
- **Why:** Separates authentication from credentials (session token vs email/password). MFA-friendly. Safe for `.gitignore`.
- **How:** Explain structure (Playwright storage format), initialization flow (interactive login), and why it's excluded from VCS.

### 5. Anti-Ban Thresholds as Configuration, Not Hardcoded
- **Why:** `ExecutionWindowController` enforces 45 profiles/day; `AnomalyDetector` watches for 403/429 responses. These need to be tunable based on LinkedIn policy changes or account age.
- **How:** Document defaults, explain how to adjust, provide metrics for monitoring.

### 6. Data Deduplication via SQLite UNIQUE Constraint on `linkedin_url`
- **Why:** Prevents duplicate profiles in DB. Insert-or-ignore semantics; no upstream logic needed.
- **How:** Show schema with UNIQUE constraint, explain "duplicate ignored" log messages, provide export query to verify coverage.

### 7. Multi-Threaded Execution with Queue-Based IPC
- **Why:** Streamlit reruns on every state change; async scrapers would block the UI. Threading + queues decouple scraper lifecycle from UI.
- **How:** Document thread spawning, message consumption, and graceful shutdown. Link to `thread_manager.py` for implementation details.

### 8. Bulk URL Processing via Text Area (Copy-Paste)
- **Why:** No CSV upload complexity; users paste 10-1000 URLs directly into Streamlit sidebar.
- **How:** Show example format, explain line-by-line parsing, warn about rate limiting across large batches.

### 9. Markdown Rendering with Standard GFM (GitHub Flavored Markdown)
- **Why:** Ensures readability on GitHub, GitLab, local editor preview, and in API docs.
- **How:** Use standard Markdown syntax (no custom HTML). Test with `grip` or GitHub's live preview.

## State Management Strategy

### Streamlit Session State
**Where:** `st.session_state` (persistent across reruns within one session)

1. **Authentication State**
   - `session_validated` (bool): session.json exists and is valid
   - `validation_timestamp` (datetime): when session was last validated
   - `validation_status` (str): "valid", "expired", "missing"

2. **Scraping State**
   - `scraper_worker` (ScraperWorker): background thread manager
   - `collection_active` (bool): scraping in progress
   - `collected_profiles` (list): profiles collected this session
   - `status_log` (list): real-time status messages

3. **Thread Communication**
   - `thread_lock` (threading.Lock): protects concurrent access to shared state
   - `last_message_check` (float): timestamp of last message queue poll
   - `current_warning` (OperationWarning): latest anomaly/warning to display

### SQLite Persistence
**Where:** `linkedin_profiles.db` (append-only, dedup via UNIQUE constraint)

- Profile inserts are idempotent (INSERT OR IGNORE)
- No manual cleanup needed; duplicates are logged and ignored
- Periodically export to CSV/Excel for backup

### Configuration Persistence
**Where:** JSON files (state, credentials)

- `session.json` — Playwright browser context storage (cookies, localStorage)
- `execution_window.json` — Daily profile count and reset history
- `.env` — Credentials and debug flags (NOT committed)

## Risks and Edge Cases

### Risk 1: PLAYWRIGHT_HOST_PLATFORM_OVERRIDE Confusion
**Symptom:** Users on Ubuntu 26.04 see "Chromium not found" or "driver mismatch"  
**Root Cause:** Playwright auto-selects ubuntu22.04 or ubuntu20.04 drivers on Ubuntu 26.04  
**Mitigation:** 
- Document with explicit warning box
- Show `export` command
- Provide validation step: `playwright show-trace` or test run
- Link to Playwright issue tracker

### Risk 2: Session Expiry During Long Scrape
**Symptom:** Scraper runs fine for 30 min, then all requests return 401  
**Root Cause:** LinkedIn session tokens expire (typically 12-24h)  
**Mitigation:**
- Document in ops section: "Session expires ~ 1 day. Restart scraper if you see 401 errors."
- UI shows "Revalidate Session" button
- Log message clearly states token is stale

### Risk 3: Rate Limit (429) After Hitting Daily Limit
**Symptom:** After 45 profiles, all subsequent requests get 429  
**Root Cause:** ExecutionWindowController enforces hard limit; LinkedIn may also rate-limit  
**Mitigation:**
- Document 45 profiles/day is conservative
- Provide tuning instructions if user wants to increase (with warning)
- Show how to check current day's count in execution_window.json
- Recommend waiting 2-4 hours before resuming

### Risk 4: Duplicate Profiles (User Perception vs. Reality)
**Symptom:** User pastes same URL twice, sees "Profile already in DB"  
**Root Cause:** UNIQUE constraint on linkedin_url; insert returns 0 rows  
**Mitigation:**
- Document: "Duplicates are silently ignored; check status log for 'Duplicate ignored' message"
- Provide SQL query to verify no actual duplicates exist: `SELECT linkedin_url, COUNT(*) FROM linkedin_profiles GROUP BY linkedin_url HAVING COUNT(*) > 1`

### Risk 5: Export File Is Empty or Incomplete
**Symptom:** User clicks "Export CSV" and gets blank file  
**Root Cause:** 
- No profiles in DB yet
- openpyxl not installed (Excel only)
- DB connection error
**Mitigation:**
- Document: "Check Database Info widget shows profile count > 0"
- Error message in app: "No profiles to export" or "openpyxl required for Excel export"
- Recommend CSV first (no external dependency)

### Risk 6: Chromium Segmentation Fault on Headless Linux
**Symptom:** `[1234:1234:0605/181234.567:ERROR] Segmentation fault (core dumped)`  
**Root Cause:** Missing X11 or Chromium runtime libraries  
**Mitigation:**
- Document: "Install dependencies: `apt-get install -y libx11-6 libxrandr2 libfontconfig1`"
- Show DISPLAY env var setup (auto-default :99)
- Link to Chromium troubleshooting guide

### Edge Case 1: URL Format Variations
**Scenarios:**
- `https://www.linkedin.com/in/username` (standard)
- `https://linkedin.com/in/username/` (trailing slash)
- `https://www.linkedin.com/in/username/` (both)
- `https://www.linkedin.com/company/acme/` (company, not profile)

**Mitigation:** Document expected format. Scraper logs invalid URLs and continues. Recommend normalizing URLs via preprocessing script (not included in README, but noted as possible enhancement).

### Edge Case 2: MFA Timeout
**Scenario:** User takes >120s to enter MFA code  
**Mitigation:** Document mfa_timeout parameter, increase if needed: `SessionManager(mfa_timeout=300000)` for 5 minutes

### Edge Case 3: Database Corruption
**Scenario:** sqlite3 file is corrupted or locked  
**Mitigation:** Document backup/restore. Provide command: `cp linkedin_profiles.db linkedin_profiles.db.backup && rm linkedin_profiles.db` (loses data, but fresh start)
