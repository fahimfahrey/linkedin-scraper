> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build robust profile data extraction layer using BeautifulSoup4 with lxml parser, leveraging high-reliability targets like JSON-LD metadata and semantic HTML containers instead of volatile CSS classes, with comprehensive fail-safe error handling to prevent single missing elements from crashing the scraping pipeline.

**Architecture:** Pair Playwright's page content retrieval with BeautifulSoup4 lxml parsing to extract structured profile data (name, headline, workplace, jobs, schools). Each extraction segment wrapped in isolated try/except closure. Multi-strategy fallback selectors prioritize JSON-LD metadata, then semantic containers (headers, landmarks), then Open Graph meta tags. Test suite validates extraction against sample profile HTML blocks with both complete and partial data.

**Tech Stack:** Playwright (page rendering), BeautifulSoup4 (HTML parsing), lxml (fast parser), pytest (unit tests), pytest-asyncio (async test support), pytest-cov (coverage tracking).

## Architecture Decisions

1. **Parser Choice**: Use `lxml` parser (BeautifulSoup skill recommends this for speed and leniency with malformed HTML). It's already in requirements.txt via beautifulsoup4.

2. **Multi-Strategy Fallback**: Each extraction function tries JSON-LD first (most reliable), then semantic selectors, then Open Graph meta tags, then DOM traversal. This prioritizes structured data over volatile UI classes.

3. **Fail-Safe Closures**: Each extraction segment (name, headline, workplace, jobs, schools) in `extract_profile_with_beautifulsoup()` wrapped in individual try/except to isolate errors. Missing headline doesn't crash job extraction.

4. **Default Values**: All extraction functions return sensible defaults (empty strings for scalars, empty lists for arrays) on total failure. Profile dict always contains all keys, never None.

5. **Async/Await**: `extract_profile_with_beautifulsoup()` is async because it calls `page.content()`. All extraction helper functions (`extract_full_name`, `extract_headline`, etc.) are sync (they only parse already-fetched HTML).

6. **Test Fixtures**: Sample HTML provides both complete profile structure and edge cases (missing sections, malformed JSON-LD, minimal data). Tests verify extraction succeeds even with partial HTML.

class TestExtractFullName:
    """Test extract_full_name() function with multiple strategies."""

    def test_extract_from_h1_tag(self, soup_complete):
        """Extract name from h1 tag (primary semantic header)."""
        result = extract_full_name(soup_complete)
        assert result == "John Smith"

    def test_extract_from_jsonld_when_h1_missing(self):
        """Fallback to JSON-LD when h1 not present."""
        html = '''
        <html>
        <head>
          <script type="application/ld+json">
          {"name": "Alice Johnson"}
          </script>
        </head>
        <body></body>
        </html>
        '''
        soup = BeautifulSoup(html, "lxml")
        result = extract_full_name(soup)
        assert result == "Alice Johnson"

    def test_extract_from_og_title_fallback(self):
        """Fallback to og:title when h1 and JSON-LD missing."""
        html = '''
        <html>
        <head>
          <meta property="og:title" content="Bob Wilson">
        </head>
        <body></body>
        </html>
        '''
        soup = BeautifulSoup(html, "lxml")
        result = extract_full_name(soup)
        assert result == "Bob Wilson"

    def test_return_default_when_all_missing(self, soup_minimal):
        """Return default value when no extraction strategy succeeds."""
        result = extract_full_name(soup_minimal)
        # Jane Doe should be in h1
        assert result in ["Jane Doe", ""]

    def test_strip_whitespace(self):
        """Extracted name is stripped of whitespace."""
        html = '<h1>  John Doe  </h1>'
        soup = BeautifulSoup(html, "lxml")
        result = extract_full_name(soup)
        assert result == "John Doe"
        assert result == result.strip()

    def test_ignore_short_strings(self):
        """Ignore names shorter than 2 characters."""
        html = '<h1>A</h1>'
        soup = BeautifulSoup(html, "lxml")
        result = extract_full_name(soup)
        assert result == ""  # Falls through to default

## State Management Strategy

**BeautifulSoup Parsing State:**
- No persistent state. Each `extract_profile_with_beautifulsoup()` call parses fresh HTML from Playwright page.
- BeautifulSoup soup object created fresh per profile.
- No caching of parsed data (profiles change, don't reuse old soup).

**Error State:**
- Extraction errors logged at DEBUG level (low priority, expected for missing sections).
- Profile dict always returned with default values, never None or partial.
- Caller (traverse_profile) always gets complete dict structure.

**Fail-Safe Mechanism:**
- Each extraction segment wrapped in individual try/except.
- If headline extraction fails, rest of profile still extracts.
- If jobs array fails to parse, schools array continues.
- No exception propagation; all errors caught and logged locally.

## Accessibility Considerations

This is a data extraction layer, not user-facing UI. No accessibility requirements apply. HTML parsing uses semantic HTML patterns (headers, sections, landmarks) which are already accessible.

## Risks and Edge Cases

### Risk 1: JSON-LD Malformation
**Scenario:** JSON-LD script tag contains invalid JSON syntax.
**Mitigation:** Wrapped in try/except with json.loads(). Fallback to semantic selectors.
**Test:** `test_malformed_jsonld_ignored` verifies fallback to h1.

### Risk 2: Missing HTML Sections
**Scenario:** Profile HTML has no experience section, jobs array fails completely.
**Mitigation:** extract_jobs_array() returns empty list []. Caller always gets valid dict.
**Test:** `test_return_empty_array_when_no_jobs` ensures empty list, not None.

### Risk 3: Nested Dictionary Access
**Scenario:** JSON-LD worksFor is object vs list vs None, causing KeyError.
**Mitigation:** Type checking (isinstance) before access. Defaults to empty string.
**Test:** `test_handle_worksFor_as_list` handles both cases.

### Risk 4: Whitespace and Encoding
**Scenario:** Profile text contains non-breaking spaces (U+00A0), unicode names.
**Mitigation:** `get_text(strip=True)` handles most; regex patterns case-insensitive.
**Test:** `test_special_characters_preserved` ensures unicode preserved.

### Risk 5: HTML Parser Inconsistency
**Scenario:** Different parsers (html.parser vs lxml) parse HTML differently.
**Mitigation:** Locked to lxml (Playwright skill recommends, robust against malformed HTML).
**Test:** `test_lxml_handles_malformed_html` auto-closes unclosed tags gracefully.

### Risk 6: Selector Bloat Over Time
**Scenario:** LinkedIn UI changes, selectors become stale.
**Mitigation:** Prioritize structured data (JSON-LD) over CSS selectors. Fallback chain means single selector failure doesn't crash pipeline.
**Test:** Multiple selectors tested independently; fallback order validated.

### Edge Case 1: Empty Profile
**HTML:** Single h1, no other data.
**Expected:** Full name extracted, all others empty/empty list.
**Test:** `test_minimal_profile_extraction`

### Edge Case 2: Duplicate Data
**HTML:** Same data in JSON-LD and semantic div.
**Expected:** JSON-LD extracted first (consistent).
**Test:** `test_extract_from_h1_tag` prioritizes strategy order.

### Edge Case 3: Very Long Strings
**HTML:** Name with 500+ characters.
**Expected:** Extracted as-is (no truncation in extraction layer).
**Test:** Not explicitly tested, but no length limit in code.
