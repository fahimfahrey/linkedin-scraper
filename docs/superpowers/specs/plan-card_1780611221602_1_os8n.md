# Session Manager Implementation Plan

## Overview

Implement `session_manager.py` to manage LinkedIn account persistence safely without hardcoding credentials. The solution will:

1. **Interactive Login Mode**: Launch Chromium in non-headless mode (`headless=False`) for manual user login with MFA support
2. **Session Export**: Intercept and export browser context (cookies, storage states, session tokens) to `session.json`
3. **Session Reuse**: Detect and inject `session.json` into headless Playwright contexts via `browser.new_context(storage_state="session.json")`
4. **Platform Awareness**: Respect system platform overrides (Ubuntu 26.04 environment variable patching)
5. **Security**: Never store credentials in code; all auth data captured from live browser interaction

This pattern follows Playwright best practices from `.agents/skills/playwright-best-practices/advanced/authentication.md`, specifically the **Storage State Reuse** and **Session Persistence** patterns.

## Architecture Decisions

### A. Session Storage Location

**Decision**: Store session state in `session.json` at project root (not `.auth/` subdirectory).

**Rationale**:
- LinkedIn scraper is a single-user tool per execution context (unlike multi-user test suites)
- Simpler naming convention; no need per-user files
- Already defined in task specification

**Implementation**: Configure path in `SessionManager.__init__()` as instance variable with default `"./session.json"`.

---

### B. Browser Launch Strategy

**Decision**: Use Playwright's `async_playwright()` context manager with Chromium engine.

**Rationale**:
- Consistent with project dependencies (`playwright` in requirements.txt)
- Supports both headless and non-headless modes
- Handles platform overrides via environment variables (e.g., `PLAYWRIGHT_HOST_PLATFORM_OVERRIDE`)

**Details**:
- Non-headless mode: `launch(headless=False)` for interactive login
- Headless mode: `launch(headless=True)` for automated scraping post-login
- Both respect `PLAYWRIGHT_HOST_PLATFORM_OVERRIDE` environment variable automatically

---

### C. Session Validation Logic

**Decision**: Before reusing a session from `session.json`, validate it by navigating to LinkedIn home and checking for a known authenticated page element.

**Rationale**:
- Sessions can expire or become invalid
- Early validation prevents downstream scraper failures
- Soft fallback: if invalid, prompt user to re-authenticate

**Implementation**:
- Check for presence of home feed element (e.g., feed container, logged-in user nav)
- Timeout: 10 seconds for navigation + validation
- If invalid: log warning, return `False`; caller decides whether to re-authenticate
- If valid: confirm cookies/storage exist, return `True`

---

### D. MFA Handling

**Decision**: Support manual MFA interaction in non-headless mode; pause execution to allow user completion.

**Rationale**:
- LinkedIn MFA is common; cannot automate without credentials
- Non-headless mode allows user to interact with any MFA prompt (TOTP, SMS, email)
- Session captured after MFA completes includes all necessary tokens

**Implementation**:
- After password entry, do NOT automatically look for home page
- Instead, wait with generous timeout (e.g., 120 seconds) for navigation to post-login page
- Use `page.wait_for_url()` with pattern matching home feed or dashboard
- User manually handles MFA in the browser window

---

### E. Storage State Export Format

**Decision**: Use Playwright's `page.context().storage_state()` method to export all cookies, localStorage, and sessionStorage to JSON.

**Rationale**:
- Playwright standard; compatible with `browser.new_context(storage_state=path)`
- Captures all session data needed for re-authentication
- JSON format is human-readable and debuggable

**Details**:
- Export includes: `cookies`, `origins` (localStorage + sessionStorage)
- Path: `./<configured_session_file>`
- Overwrite existing file on each successful login (stateless)
- Never commit to git (add to `.gitignore`)

### Step 2: Implement Interactive Login Method

**Method**: `SessionManager.interactive_login()`

**Logic**:
1. Call `launch_browser_for_login()` to create non-headless browser
2. Navigate to `https://www.linkedin.com/login`
3. Print message to user: "Please log in manually. You have [TIMEOUT] seconds. MFA will be handled here."
4. Call `wait_for_successful_login()` with:
   - Initial wait: password entry + MFA (up to 120 seconds)
   - Validation element: logged-in page structure (e.g., feed container, user nav)
5. On success, call `export_session()` to save state
6. Call `cleanup()` to close browser
7. Return `True` on success, `False` on timeout/failure

**Exception Handling**:
- Timeout: log warning, return `False`
- Browser crash: log error, return `False`
- No element found: assume MFA in progress, extend timeout to 120s

### Step 3: Implement Session Export Method

**Method**: `SessionManager.export_session()`

**Logic**:
1. Assert context exists and has an active page
2. Call `self.page.context().storage_state(path=str(self.session_file))`
3. Verify file was created and is not empty
4. Log success: "Session exported to session.json"
5. Return `True`

**Error Handling**:
- If export fails: log error, return `False`
- If file empty or malformed: return `False`

### Step 4: Implement Session Loading Method

**Method**: `SessionManager.load_session()`

**Logic**:
1. Check if `session.json` exists
2. If not: return `None` or raise `FileNotFoundError`
3. If yes:
   - Launch headless browser
   - Create context with `storage_state="session.json"`
   - Return context
4. On error: log and re-raise

**Code outline**:
```python
async def load_session(self) -> BrowserContext:
    if not self.session_exists():
        raise FileNotFoundError(f"Session file not found: {self.session_file}")
    
    async with async_playwright() as p:
        self.browser = await p.chromium.launch(headless=True)
        context = await self.browser.new_context(
            storage_state=str(self.session_file)
        )
        return context
```

### Step 5: Implement Session Validation Method

**Method**: `SessionManager.validate_session()`

**Logic**:
1. Check if `session.json` exists
2. Load it into a temporary context (non-headless for visibility if needed)
3. Navigate to `https://www.linkedin.com/home`
4. Wait for authenticated element (with 10s timeout):
   - Look for: feed container, user profile nav, or logout button
   - Pattern: `page.locator("[data-test-id='feed']")` or similar
5. If found: session valid, return `True`
6. If not found: session invalid, return `False`
7. Clean up temporary context

**Element Detection**:
- Use CSS selectors or XPath for LinkedIn-specific elements
- Fallback: check page title contains "LinkedIn" (unlikely to be reliable)
- Primary method: check for feed-related DOM elements

### Step 6: Implement Unified Entry Point

**Method**: `SessionManager.get_authenticated_context()`

**Logic**:
```
1. If session.json exists:
   a. Try validate_session()
   b. If valid: return load_session()
   c. If invalid: log warning, proceed to step 2
2. If session doesn't exist or is invalid:
   a. Call interactive_login()
   b. If success: return load_session()
   c. If failure: raise AuthenticationError
```

**Error Cases**:
- No session + failed login: raise `AuthenticationError("Interactive login failed")`
- Session exists but invalid + failed re-login: raise `AuthenticationError`

## State Management Strategy

### Session Lifecycle

1. **No Session File**: User runs scraper → `get_authenticated_context()` → interactive login → export session → headless context ready
2. **Valid Session File**: User runs scraper → `get_authenticated_context()` → validate → load → headless context ready (fast path)
3. **Expired Session File**: User runs scraper → validate fails → interactive login → export → headless context ready
4. **Manual Re-auth**: Delete `session.json` → next run forces interactive login

### State Variables

- `self.session_file` (Path): location of session JSON file
- `self.browser` (Browser): active Playwright browser instance
- `self.context` (BrowserContext): active browser context
- `self.page` (Page): active page object
- `self.logger` (Logger): logging instance

### Context Manager Option (Advanced)

**Optional Enhancement** (implement if time allows):
```python
async with SessionManager() as session_mgr:
    context = await session_mgr.get_authenticated_context()
    # Use context...
```

This ensures cleanup happens automatically.

## Risks and Edge Cases

### Risk 1: Session Expiration During Scrape

**Scenario**: Session loaded but expires mid-scrape.

**Mitigation**:
- Implement session refresh check at scraper start
- Add optional periodic validation during long scrapes
- Log 401/403 errors clearly for user intervention

**Implementation**: Add `SessionManager.refresh_if_needed()` method.

---

### Risk 2: MFA Timeout Too Short

**Scenario**: User is slow with MFA entry; browser closes before completion.

**Mitigation**:
- Default timeout: 120 seconds (ample for manual MFA)
- Make timeout configurable in `SessionManager.__init__()`
- Print countdown messages at 60s, 30s, 10s marks

**Implementation**: Add `mfa_timeout: int = 120000` parameter.

---

### Risk 3: Session File Corruption

**Scenario**: `session.json` exists but is malformed JSON or missing required keys.

**Mitigation**:
- Validate JSON structure in `load_session()`
- Try-except on JSON parsing
- If invalid: log error, delete corrupted file, prompt re-login

**Implementation**:
```python
try:
    with open(self.session_file) as f:
        json.load(f)
except (json.JSONDecodeError, KeyError) as e:
    logger.error(f"Corrupted session file: {e}")
    self.session_file.unlink()  # Delete
    return False
```

---

### Risk 4: Platform Override Not Respected

**Scenario**: `PLAYWRIGHT_HOST_PLATFORM_OVERRIDE` is set but Playwright ignores it.

**Mitigation**:
- Pass environment variable explicitly to `launch()` if needed
- Log detected platform at startup
- Document requirement in README

**Implementation**: Playwright auto-respects env var; no explicit code needed.

---

### Risk 5: LinkedIn Page Structure Changes

**Scenario**: Page elements used for validation (e.g., `[data-test-id='feed']`) are renamed/removed.

**Mitigation**:
- Use multiple fallback selectors
- Log element not found; allow manual override (user confirms login happened)
- Update selectors when LinkedIn structure changes

**Implementation**: Create `_find_auth_element()` helper with multiple selectors:
```python
selectors = [
    "[data-test-id='feed']",  # Primary
    "main.global-nav-container",  # Fallback
    "a[href*='/home']",  # Tertiary
    "button[aria-label*='Profile']"  # Last resort
]
for selector in selectors:
    try:
        await page.wait_for_selector(selector, timeout=5000)
        return True
    except:
        continue
return False
```

---

### Risk 6: Concurrent Access (Multiple Processes)

**Scenario**: Two scraper instances try to use/modify `session.json` simultaneously.

**Mitigation**:
- Add file lock mechanism (optional, for now: assume single-user per machine)
- Document: run one scraper instance at a time
- Future: implement file locking with `filelock` package if needed

---

### Risk 7: Browser Resource Leak

**Scenario**: Exception in interactive login leaves browser open.

**Mitigation**:
- Use try-finally or context manager to ensure cleanup
- Test cleanup in exception paths

**Implementation**: Wrap interactive login in try-finally:
```python
try:
    # interactive login code
finally:
    await self.cleanup()
```

- **Element Detection**: LinkedIn's DOM is complex. Use multiple selectors and be prepared to update them.
- **Testing**: Full end-to-end test of real LinkedIn login is optional (may require real account). Mock-based tests are sufficient.
- **Security**: Never log session tokens, cookies, or credentials. Log only success/failure states.
