1. **Strict daily profile limit enforcement** (40–50 profiles/day) with automatic termination when exceeded
2. **Enhanced anomaly detection** for platform redirects, forced logouts, CAPTCHA intervention, and rate-limit responses
3. **Graceful emergency shutdown protocol** that flushes collected profiles to SQLite, logs system state, and cleanly kills browser processes

The implementation prioritizes safety-first design: if any ambiguity arises, the scraper terminates rather than continue. All state persists to database before shutdown, ensuring no data loss.

## Architecture Decisions

### 1. **Fail-Safe Defaults**
- All ambiguous states → emergency shutdown (conservative)
- If anomaly detection is uncertain, shut down rather than continue
- Rationale: Account security is non-negotiable; lost execution time is acceptable

### 2. **Layered Anomaly Detection**
- **URL analysis:** Pattern matching on known/suspicious redirects
- **HTML content analysis:** CAPTCHA/login regex patterns (precompiled for performance)
- **HTTP response codes:** Rate-limit signatures (429, 503)
- **Rate-limit heuristics:** Consecutive rapid failures trigger incident
- Rationale: Redundancy ensures no single detection vector is a single point of failure

### 3. **Execution Window Enforcement**
- Hard limit: 40–50 profiles/day (default 45)
- Soft warning at 80% (36 profiles)
- Critical warning at 90% (40 profiles)
- Hard kill at 100% (45 profiles)
- Rationale: Graduated warnings give visibility; hard kill prevents runaway collection

### 4. **Emergency Shutdown Protocol**
Sequence (order critical):
1. **Signal:** Catch `ExecutionWindowExceeded` or `SecurityIncidentDetected`
2. **Flush:** Write all collected profiles to SQLite (in-memory buffer → DB)
3. **Log:** Serialize system state snapshot (timestamp, count, anomaly_type, memory, browser URL, etc.)
4. **Close:** Call `safe_browser_cleanup()` to close browser + context
5. **Kill:** Terminate browser subprocess cleanly
6. **Exit:** Emit `ExecutionComplete` with `success=False` + incident details

Rationale: Database flush happens before cleanup to guarantee no data loss. State log preserves forensic context for incident review.

### 5. **Browser Process Management**
- Use existing `safe_browser_cleanup()` function (already in scraper.py)
- Wrap in try/except; log cleanup failures but do not block shutdown
- Ensure browser subprocess is terminated even if context/page close fails
- Rationale: Prevent zombie processes; clean resource release

### 6. **State Persistence**
- `ExecutionWindowController` persists to `execution_window.json` after every profile check
- System state snapshots written to `security_incidents.log` (JSON Lines format) on emergency shutdown
- All profiles flushed to SQLite before process exit
- Rationale: Enables forensic analysis and prevents data loss

## State Management Strategy

### **Persistent State**

1. **Execution Window State** (`execution_window.json`)
   - Tracks daily profile count, date, reset count
   - Updated atomically after each profile check
   - Survives process restarts (enables resumption)

2. **Security Incident Log** (`security_incidents.log`)
   - JSON Lines format (one incident per line)
   - Appended on emergency shutdown
   - Fields: timestamp, profiles_collected, anomaly_type, browser_url, memory_usage_mb

3. **Profile Database** (`linkedin_profiles.db`)
   - SQLite with unique constraint on `linkedin_url`
   - Flushed before emergency shutdown
   - Enables forensic verification of collected data

### **In-Memory State**

1. **profiles_collected** (list)
   - Buffered profiles before DB flush
   - Cleared after emergency shutdown

2. **url_history** (deque, max length 10)
   - Tracks recent page URLs for pattern analysis
   - Enables detection of redirect chains

3. **failure_count** (int)
   - Tracks consecutive page load failures
   - Reset on successful load

## Risks and Edge Cases

### **Risk 1: Database Flush Timeout**
- **Scenario:** Flush takes >30s due to large profile count
- **Mitigation:** Add async timeout wrapper; log partial flush if timeout
- **Impact:** Medium — data loss possible

### **Risk 2: Zombie Browser Processes**
- **Scenario:** Browser cleanup raises exception; subprocess not killed
- **Mitigation:** Explicit subprocess termination in finally block
- **Impact:** High — resource leak

### **Risk 3: False-Positive Anomaly Detection**
- **Scenario:** Legitimate page matches CAPTCHA pattern
- **Mitigation:** Require multiple pattern matches for incident
- **Impact:** Low — conservative bias is preferred

### **Risk 4: State File Corruption**
- **Scenario:** `execution_window.json` corrupted mid-write
- **Mitigation:** Atomic write with temp file + rename
- **Impact:** Medium — loss of daily count

### **Risk 5: Race Condition in URL Checking**
- **Scenario:** Multiple workers increment counter simultaneously
- **Mitigation:** File-based locking or queue-based synchronization
- **Impact:** Medium — potential limit bypass

### **Edge Case 1: Daily Rollover at Midnight**
- **Mitigation:** Check date before incrementing
- **Impact:** Low — handled by `reset_if_new_day()`

### **Edge Case 2: Page Load Partial Failure**
- **Mitigation:** Wrap page access in try/except
- **Impact:** Low — graceful fallthrough

### **Edge Case 3: Browser Crash During Shutdown**
- **Mitigation:** Wrap cleanup in try/except; continue to next step
- **Impact:** Low — does not block flush
