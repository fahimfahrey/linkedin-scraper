# Anti-Ban Guardrails Implementation Plan

**Objective:** Enhance `scraper.py` with strict programmatic protective measures to prevent account freezes and enforce daily profile limits (40-50 max), detect anomalies (redirects, logout, CAPTCHA), and gracefully handle security incidents with safe shutdown.

**Scope:** Modify `scraper.py` only. Add new protective classes/functions. Test with artificial break conditions. Stage locally with specified commit message.

---

## Overview

LinkedIn aggressively monitors scraper activity. Commercial-scale collection triggers rate limiting, account restrictions, or permanent bans. This plan implements layered defenses:

1. **ExecutionWindowController**: Tracks profiles processed per day. Hard stops execution at threshold (40–50 profiles).
2. **AnomalyDetector**: Intercepts platform security signals (CAPTCHA, redirects, forced logout).
3. **SecurityIncidentHandler**: Catches anomalies, dumps working data to `database.py`, logs state, kills browser cleanly.
4. **Integration points**: Wrap main loop and critical page transitions with these guards.

Expected outcome: Scraper runs safely within daily limits, detects attacks preemptively, exits without data loss.

## Architecture Decisions

### 1. **ExecutionWindowController Design**
   - **Strategy**: Track profiles via counter + persist to JSON state file (default: `execution_window.json`)
   - **Threshold**: Configurable, default 45 profiles/day
   - **Reset logic**: New file created daily (timestamped). Check age, reset if >24h old.
   - **Thread-safety**: Not required (async-only, single process). Use `asyncio.Lock` for shared state if needed later.
   - **Alternative considered**: Database table (rejected: adds SQLite dependency, slower recovery on crash)

### 2. **AnomalyDetector Implementation**
   - **Method**: Regex + DOM inspection on every page response
   - **Signals monitored**:
     - CAPTCHA: CSS selector `[id*='recaptcha'], [id*='challenge'], iframe[src*='recaptcha']`
     - Forced logout: URL contains `/login` OR page title contains "Sign in"
     - Redirect loops: Track `page.url` against known LinkedIn paths; deviation = anomaly
     - Rate limit page: Text matching `(rate limited|try again|too many requests|slow down)`
   - **Alternative considered**: Use Playwright's request interception (rejected: overhead, fragile with async)

### 3. **SecurityIncidentHandler Flow**
   ```
   Anomaly detected
     ↓
   SecurityIncidentDetected exception raised
     ↓
   catch in main loop / scrape_profile()
     ↓
   dump_emergency_state() to database
     ↓
   log incident details (timestamp, type, profiles processed)
     ↓
   safe_browser_cleanup()
     ↓
   re-raise as graceful exit (exit code 0 for task scheduling)
   ```

### 4. **Logging Strategy**
   - Use existing `logger` instance (already defined in scraper.py)
   - Add new log level entries:
     - `logger.warning("Daily limit approaching: X/45 profiles")` at 35+ profiles
     - `logger.critical("SECURITY INCIDENT: CAPTCHA detected. Emergency shutdown initiated.")` on anomaly
     - `logger.info("Execution window reset: new day detected")` daily
   - Audit log (JSON): `execution_audit.json` with entries `{timestamp, event, profiles_count, anomaly_type}`

### 5. **State Persistence**
   - **execution_window.json**: `{date: "2026-06-05", count: 23, reset_count: 0}`
   - **execution_audit.json**: Array of `{timestamp: ISO8601, event: "profile_processed|threshold_warning|anomaly_detected", details: {...}}`
   - Both stored in project root (alongside `linkedin_profiles.db`)

### 6. **Error Recovery**
   - If `execution_window.json` is malformed, reinitialize (assumes fresh day)
   - If anomaly detection fails (regex error), log and continue (don't crash on detection logic)
   - If browser cleanup fails, attempt `process.kill()` (OS-level SIGTERM)

## State Management Strategy

### Execution State Lifecycle

```
Session Start
  ↓
Load execution_window.json (or create if missing)
  ↓
Check if today's date matches stored date
  ├─ If yes: use existing counter
  └─ If no: reset counter, update date
  ↓
For each profile:
  ├─ Increment counter
  ├─ Check if counter >= max (45)
  │  └─ If yes: raise ExecutionWindowExceeded
  └─ Scrape profile
  ↓
Session End → Save state to execution_window.json
```

### Anomaly Detection Flow

```
After page.goto() success:
  ↓
Page loaded (networkidle)
  ↓
AnomalyDetector.inspect_page(page)
  ├─ Get page.content()
  ├─ Get page.url
  ├─ Get page.title
  ├─ Run all regex + DOM checks in parallel (or sequence)
  └─ Return anomaly_type or None
  ↓
If anomaly detected:
  └─ Raise SecurityIncidentDetected
```

### Emergency Dump Format

```json
{
  "timestamp": "2026-06-05T14:23:45.123456",
  "incident_type": "captcha",
  "profiles_processed": 23,
  "profiles_buffer": [
    {"linkedin_url": "...", "full_name": "...", ...},
    ...
  ],
  "execution_state": {
    "max_per_day": 45,
    "counter": 23,
    "daily_reset": "2026-06-05"
  },
  "system_state": {
    "page_url": "https://...",
    "page_title": "...",
    "browser_version": "...",
    "hostname": "..."
  }
}
```

## Risks and Edge Cases

### Risk 1: Daily Reset Race Condition
**Risk**: Multiple processes reading/writing `execution_window.json` simultaneously.
**Mitigation**: Use `asyncio.Lock` for file I/O within process. Document as single-process per session.

### Risk 2: Anomaly Detector False Positives
**Risk**: Regex patterns might match legitimate LinkedIn content (e.g., ad with "rate limited" text).
**Mitigation**: Combine multiple signals (URL + content + title). Log all detections for review.

### Risk 3: Browser Process Lingering After Crash
**Risk**: `safe_browser_cleanup()` fails; orphaned Chromium processes consume resources.
**Mitigation**: Use `psutil` to find + kill by process name or parent PID. Document manual cleanup if needed.

### Risk 4: Malformed execution_window.json
**Risk**: Corrupted state file blocks startup.
**Mitigation**: Try/except JSON parse; on failure, reinitialize with fresh state and log warning.

### Risk 5: Emergency Dump Overwrites Previous Incident
**Risk**: Two incidents in same second; second dump overwrites first.
**Mitigation**: Use microsecond precision in timestamp; include random suffix if collision detected.

### Edge Case 1: Counter at Exactly Threshold (45)
**Intended**: Reject 45th profile (stop at 44).
**Implementation**: `if counter >= max_profiles: raise` (not `>`).

### Edge Case 2: Page Load Times Out Before Anomaly Check
**Intended**: `traverse_profile()` raises RuntimeError on timeout; not caught by incident handler.
**Acceptable**: Incident handler only catches known anomalies. Timeouts are separate concern.

### Edge Case 3: Profile Extraction Succeeds After Anomaly Detection
**Scenario**: Page partially loaded; anomaly detected but some data extractable.
**Intended**: Dump all buffered profiles; don't attempt extraction post-anomaly.
**Implementation**: Raise exception immediately on anomaly; catch in main loop.

### Edge Case 4: Nested Exception in Browser Cleanup
**Scenario**: `safe_browser_cleanup()` raises while handling SecurityIncidentDetected.
**Intended**: Log cleanup exception; exit gracefully.
**Implementation**: Wrap cleanup in try/finally; never re-raise from cleanup.

### Regex Patterns (Suggested)
```python
# CAPTCHA detection
re.compile(r"recaptcha|hcaptcha|challenge", re.I)

# Forced logout
URL pattern: /login, /signup
Title pattern: "Sign in", "Log in"

# Rate limit
re.compile(r"rate.?limit|too.?many|slow.?down|try.?again", re.I)
```

### State File Locations
All state files in project root (alongside `linkedin_profiles.db`):
- `execution_window.json` — daily counter + reset tracking
- `execution_audit.json` — event log (append-only)
- `emergency_dump_TIMESTAMP.json` — created only on incident

### Graceful Exit Semantics
- Exit code 0: Normal completion OR security incident (treated as intentional halt, not error)
- Exit code 1: Unhandled exception (retryable by scheduler)
- Exit code 2: Configuration error (not retryable)
